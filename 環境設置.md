## 二、正式環境邊界

本 repo 的環境分成兩層。

### 2.1 系統級環境

系統級環境指：

- Windows
- Visual Studio / MSVC
- Windows SDK / UCRT
- CUDA Toolkit
- Ninja
- COLMAP 二進位

這一層目前的原則是：

- **保留既有 CUDA 12.8，不動系統安裝**
- 不在 Python 虛擬環境內處理 Visual Studio / Windows SDK 問題
- COLMAP 視為 repo 內附帶的外部工具，而不是 Python 套件

### 2.2 工作區 Python 環境

Python 層只處理：

- `.venv`
- `pip` 套件
- `torch / torchvision / torchaudio / gsplat`
- 由 Python 觸發的 JIT / CUDA extension 編譯

這一層目前的正式策略是：

- **舊 `.venv` 不原地修補**
- **直接重建乾淨 `.venv`**
- 所有 Python 相依與快取問題，都在新 `.venv` 中重新驗證

## 六、環境變數原則

### 6.1 這份文件不再硬寫歷史結論

舊文件中曾出現多組 `TORCH_CUDA_ARCH_LIST`、`CUDA_HOME`、SDK workaround 與 patch。  
這些內容已證明包含歷史漂移，因此不再作為這份文件的正式主體。

### 6.2 現階段的正式原則

- `CUDA_HOME` 應指向系統中的 CUDA 12.8
- 不以歷史 patch 版本作為正式預設
- 不把某一次失敗或暫時成功的 `TORCH_CUDA_ARCH_LIST` 直接寫死成正式答案
- 若 shell 內殘留 `TORCH_CUDA_ARCH_LIST=12.0`，應先清掉再安裝或訓練

### 6.3 目前已知的環境變數風險

目前工作區裡已有舊腳本曾寫入：

- 舊的 `TORCH_CUDA_ARCH_LIST`
- 舊的 SDK / INCLUDE / LIB workaround
- 舊的輸出目錄路徑假設

因此在正式重建 `.venv` 前，不能把任何單一歷史腳本視為唯一真相。

目前已知最需要避免的是：

- `TORCH_CUDA_ARCH_LIST=12.0`

這組舊設定曾導致 `Unknown CUDA arch (120)`，不應再沿用。

另外，這台機器上還觀察到一個系統級特性：

- 單跑 `vcvars64.bat` 不一定會自動把完整 Windows SDK / UCRT 路徑帶進來

因此若 3DGS 訓練仍出現 `corecrt.h` 類錯誤，應優先檢查同一個 session 內是否真的有：

- Windows SDK include 路徑
- UCRT include / lib 路徑
- SDK bin 路徑
