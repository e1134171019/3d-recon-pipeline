## 三、舊 `.venv` 汙染或安裝殘留

### 症狀

- `pip` 出現 `Ignoring invalid distribution ~orch`
- `site-packages` 內看到異常殘留，例如 `~unctorch`
- 同一份文件和實際安裝版本對不上
- 環境曾經歷多次失敗安裝 / 重裝

### 原因

這代表舊 `.venv` 已經不是乾淨基線。  
對這類問題，正式策略不是局部修補，而是整個重建。

### 正式解法

```powershell
cd C:\3d-recon-pipeline
Remove-Item .venv -Recurse -Force
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip setuptools wheel
```

### 不建議做的事

- 在壞掉的舊 `.venv` 上反覆 `pip install --force-reinstall`
- 手動刪一半 `site-packages` 後繼續疊安裝
- 把歷史 log 裡的 workaround 當成環境基線

## 四、系統工具找不到

### 4.1 `python` 找不到

#### 症狀

- `python --version` 失敗
- `python -m venv .venv` 失敗

#### 解法

先安裝 Python 3.12，並確認命令列可直接呼叫 `python`。

---

### 4.2 `nvcc` 找不到

#### 症狀

- `nvcc --version` 失敗

#### 解法

這表示系統級 CUDA 工具鏈沒進 PATH，或 CUDA Toolkit 未完整安裝。  
本 repo 的正式前提是：

- 系統已有 CUDA 12.8
- 本輪不更動系統 CUDA 版本

先修好系統 PATH，再進 Python 安裝。

---

### 4.3 `ninja` 找不到

#### 症狀

- `ninja --version` 失敗
- `gsplat` 編譯時提示缺少 `ninja`

#### 解法

安裝 Ninja，並確認它可直接從命令列執行。

---

### 4.4 `cl.exe` 找不到

#### 症狀

- `where cl.exe` 找不到
- CUDA extension 編譯時說缺少 C++ compiler

#### 解法

確認 Visual Studio 2022 C++ 工具鏈已安裝。  
這是系統級工具問題，不在 `.venv` 內解決。

### 症狀

- `torch.cuda.is_available()` 為 `False`
- `torch` 可以 import，但看不到 GPU
- `test_cuda.py` 檢查失敗

### 先做的最小檢查

### 正式處理原則

1. 先重建乾淨 `.venv`
2. 再做最小驗證
3. 不要先套歷史 `TORCH_CUDA_ARCH_LIST` workaround

## 八、`gsplat` JIT / CUDA extension 失敗

### 症狀

- `import gsplat` 可成功，但訓練一開始就炸
- 訓練時出現 `csrc`、`_backend`、`_jit_compile`、`Error building extension`
- 日誌中出現 Windows SDK / UCRT / `corecrt.h` / CUDA arch 相關錯誤
- 出現 `Unknown CUDA arch (120)`

### 這一類問題的本質

這不是單純的 Python 套件安裝失敗，而是：

- Python 層
- CUDA extension 編譯
- Windows SDK / UCRT
- MSVC / Ninja

一起參與的工具鏈問題。

### 目前已驗證的常見 Windows 卡點

這個工作區在 2026-04-08 實際踩到並驗證過以下幾類錯誤：

1. `corecrt.h` 找不到
   - 原因：訓練所在的 session 沒有帶完整 Windows SDK / UCRT include/lib 路徑

2. `bool char` / `CUDACachingAllocator.h` 之類的奇怪 C++ 錯誤
   - 原因：Windows SDK `rpcndr.h` 的 `#define small char` 污染了 Torch header 的參數名

3. `cl : command line error D8021 : invalid numeric argument '/Wno-attributes'`
   - 原因：`gsplat` 的 backend 在 Windows 上仍帶入 Linux 風格的 `-Wno-attributes`

4. `Could not find a backend to open ... traj_*.mp4`
   - 原因：缺少 `imageio` 的 MP4 backend
   - 解法：安裝 `imageio-ffmpeg`（目前已寫入 `requirements.txt`）

### 正式排查順序

1. 驗證系統工具

```powershell
python --version
nvcc --version
ninja --version
where cl.exe
```

2. 驗證 Python 層

```powershell
python -c "import torch; print(torch.__version__); print(torch.cuda.is_available())"
python -c "import gsplat; print(gsplat.__version__)"
python test_cuda.py
```

3. 檢查環境變數

```powershell
$env:TORCH_CUDA_ARCH_LIST
$env:CUDA_HOME
```

### 目前文件策略

本文件不再把歷史 patch 直接列成正式步驟，例如：

- 手改 `cpp_extension.py`
- 手改 `gsplat` 套件內部檔案
- 直接套用舊腳本中的 `TORCH_CUDA_ARCH_LIST`

這些只會在後續正式版本矩陣與安裝腳本收斂後，按需要重新驗證。

### 已知特定錯誤：`Unknown CUDA arch (120)`

這通常表示 shell 內還殘留舊的：

- `TORCH_CUDA_ARCH_LIST=12.0`

正式處理方式：

PowerShell：

```powershell
Remove-Item Env:TORCH_CUDA_ARCH_LIST -ErrorAction SilentlyContinue
```

CMD：

```bat
set TORCH_CUDA_ARCH_LIST=
```

清掉後重新啟用 `.venv`，再跑：

```powershell
python test_cuda.py
```

## 十、SfM 成功，但 3DGS 沒有 checkpoint

### 症狀

- `outputs/reports/pointcloud_validation_report.json` 存在
- `can_proceed_to_3dgs = true`
- 但 `outputs/3DGS_models/ckpts/` 沒有檔案

### 代表什麼

通常表示：

- SfM 主線已經通過
- 問題出在 3DGS 啟動或 `gsplat` CUDA extension 階段

### 正式排查順序

1. 確認驗證報告存在
2. 確認 `imgdir` 和 SfM 使用的影像集一致
3. 驗證 `torch` / `gsplat` / `test_cuda.py`
4. 再看 3DGS 輸出目錄是否只留下 `cfg.yml` / `tb` 而沒有 checkpoint

如果只有 `cfg.yml` 或 tensorboard event，而沒有 `ckpt_*.pt`，多半是訓練在早期初始化或第一個 step 前就失敗。

### 症狀

### 正式處理原則

只有 smoke test 通過後，才進完整訓練。
